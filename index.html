Nosso modelo

Ownership com conhecimento circulando: quem criou o sistema segue cuidando dele — trabalho em equipe, com individualidade. E cada sistema ganha backup e documentação, para que o conhecimento nunca fique preso em uma só pessoa.

01
Por que mudar

De onde viemos, para onde vamos

Construir tudo internamente é uma força do Instituto — mas o que nos trouxe até aqui não sustenta o próximo nível sem organização.

Hoje

  • Código espalhado em contas pessoais do GitHub (Igor e Alex)
  • Demandas chegam por qualquer canal, sem registro nem priorização
  • Conhecimento concentrado em quem criou cada sistema
  • Sem ambientes definidos: mudanças vão direto para produção
  • Documentação inexistente ou informal

A partir de agora

  • GitHub organizacional do Instituto CDT: tudo centralizado e versionado
  • Desenvolvimento acelerado por IA (Claude Code e Lovable) sobre uma base organizada
  • Canal único de demandas, com triagem e priorização
  • Cada sistema com dono principal + backup, e documentação padrão
  • Fluxo dev → homologação → produção em todos os projetos
  • Setor preparado para receber e formar novos devs

    Como o trabalho chega e sai

    Fluxo de demandas

    Toda demanda passa a ter um caminho único: entra registrada, é triada, priorizada e entregue com homologação. Escolha o tipo de demanda para ver o fluxo.

    1

    Solicitação registrada

    O setor solicitante abre a demanda no canal oficial de TI (board no Trello), descrevendo objetivo e resultado esperado. Nada entra "por corredor".

    SOLICITANTE
    2

    Triagem e alinhamento

    Igor avalia esforço, impacto e dependências. Demandas grandes são validadas com a Evelin antes de entrar na fila.

    IGOR · EVELIN
    3

    Priorização

    Na reunião do setor, a fila é priorizada e a demanda é atribuída ao dev responsável, considerando ownership e carga de trabalho.

    EQUIPE TI
    4

    Desenvolvimento com versionamento

    Trabalho em branch no GitHub da CDT, com Pull Request e revisão de código por outro dev antes de mesclar.

    DEV RESPONSÁVEL
    5

    Homologação com o solicitante

    A entrega é validada em ambiente de homologação por quem pediu. Só sobe para produção com aprovação registrada.

    SOLICITANTE · DEV
    6

    Produção + documentação

    Deploy em produção, atualização da documentação do projeto e comunicação de encerramento ao solicitante.

    DEV RESPONSÁVEL
    1

    Chamado aberto no canal de suporte

    Ajustes e correções entram como chamados, com descrição do problema e print/exemplo quando possível.

    SOLICITANTE
    2

    Classificação

    Igor classifica: bug, ajuste rápido ou melhoria. Chamados simples são ótima porta de entrada para o Dev Júnior, com supervisão.

    IGOR
    3

    Atendimento pelo dono do sistema (ou backup)

    O dono principal atende; se estiver indisponível, o backup assume usando a documentação do projeto.

    DONO · BACKUP
    4

    Validação e fechamento

    Correção validada com o solicitante e chamado encerrado com registro do que foi feito.

    DEV · SOLICITANTE
    1

    Comunicação imediata

    Sistema fora do ar ou erro crítico: aviso direto ao Igor (ou Alex na ausência), fora da fila normal.

    QUALQUER PESSOA
    2

    Resposta prioritária

    O incidente passa na frente de tudo. Dono do sistema atua na correção; Igor comunica status à Evelin e aos afetados.

    DONO · IGOR
    3

    Pós-incidente

    Depois de resolvido: registro da causa, o que foi feito e o que muda para não repetir. Aprendizado vira documentação.

    05
    Fundação técnica

    GitHub organizacional da CDT

    Hoje os projetos vivem nas contas pessoais do Igor e do Alex. Criar a organização instituto-cdt no GitHub é o primeiro passo — é o que torna todo o resto possível, inclusive o nosso jeito de desenvolver com IA.

    📦

    Centralização

    Todos os repositórios migrados para a organização da CDT. O código é patrimônio do Instituto, com acesso controlado por papel.

    🔀

    Versionamento e revisão

    Branches por demanda, Pull Requests obrigatórios e revisão por outro dev antes de mesclar. Histórico completo de tudo que mudou.

    🚀

    Homologação e deploy

    Ambientes separados e caminho único até produção. Nada sobe sem passar por homologação.

    ✨

    Nosso jeito de desenvolver: IA no centro

    No Instituto CDT, o desenvolvimento é feito todo (ou quase todo) com auxílio de IA — Claude Code e Lovable. O GitHub organizacional é o que faz esse modelo funcionar em equipe: cada dev clona o projeto, trabalha localmente com o Claude Code e sobe o resultado na branch correta. Todos aceleram com IA, e o projeto permanece único, atualizado e versionado.

    Regra do setor: todo repositório terá um arquivo CLAUDE.md descrevendo o projeto — stack, convenções, como rodar, o que não tocar. É a documentação que a IA lê: qualquer dev abre qualquer projeto e o Claude Code já trabalha seguindo as regras da casa.

    clona o repositório da CDT→ desenvolve local com Claude Code / Lovable→ push na branch da demanda→ Pull Request + revisão
    desenvolvimento→ homologação→ produção
    Padrão mínimo por repositório: README com o que o sistema faz e como rodar localmente · CLAUDE.md com stack, convenções e regras do projeto (a documentação que a IA lê) · guia de deploy · variáveis de ambiente documentadas · registro de decisões importantes. É esse padrão que permite qualquer dev — humano ou com IA — assumir qualquer projeto.

    Regra de ouro: nenhum conhecimento pode existir só na cabeça de uma pessoa. Se só um dev consegue mexer num sistema, numa automação ou num acesso, isso é um risco para o Instituto — a documentação é o nosso seguro.
    07
    Novo dev

    Onboarding do Dev Júnior

    Entrada progressiva, sob liderança direta do Igor: primeiro conhecer, depois atuar com supervisão, depois ganhar autonomia. Cada fase só avança quando a anterior estiver consolidada.

    Conhecer o Instituto, os sistemas e as pessoas. Configurar o ambiente de desenvolvimento usando a documentação — e melhorá-la no processo: o olhar de quem chega encontra as lacunas.
    • Acesso ao GitHub da CDT com permissões de júnior
    • Tour guiado pelo ecossistema com Igor e Alex
    • Rodar os projetos localmente seguindo o README
    Atuar nos chamados de suporte de baixa criticidade, sempre com Pull Request revisado pelo Igor ou Alex antes de qualquer mescla.
    • Ajustes visuais, textos, correções pequenas
    • Todo código passa por revisão — sem exceção
    • Regra do PR: saber explicar o que cada mudança faz
    • 1:1 recorrente com Igor para acompanhamento
    Assumir demandas completas de menor porte, do desenvolvimento à homologação, com acompanhamento próximo.
    • Primeira feature de ponta a ponta
    • Participa da homologação com o solicitante
    • Atende chamados com mais autonomia
    Vira backup de um sistema de menor criticidade e passa a ter fila própria de demandas.
    • Backup oficial de um sistema
    • Fila própria de demandas e chamados
    • Avaliação de evolução com Igor e Evelin
    IA acelera, mas não substitui entendimento: com Claude Code e Lovable, um júnior produz muito código rápido — e código que ele não entende é dívida disfarçada de produtividade. Por isso a regra: no Pull Request, o dev precisa saber explicar o que cada mudança faz. O review existe para ensinar, não só aprovar.

    A revisão de código continua como prática permanente do time — para todos, não só para o júnior.
